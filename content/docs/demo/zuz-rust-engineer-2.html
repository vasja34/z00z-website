The Goal

Purpose – deliver a single-binary Rust daemon (“agent”) that

  1. fetches immutable Vault objects from IPFS,
  1. loads a Lua-byte-code “ticket” bundled with each vault,
  1. executes the ticket in a deterministic, capability-restricted Lua sandbox,
  1. performs all network coordination with fellow agents (FROST signing,) via I2P network
  1. submits the resulting SUI-Move call to the (stub)

    ZUZ-smart-contract on a Sui devnet.

All communication with IPFS must go through I2P.
I2P is already implemented in Docker.

The task is to simulate multiple agents that communicate with each other via I2P and treat the objects called Vaults.

Each agent implemented in Rust and runs a Lua sandboxed ByteCode program (stored in Vault), and at the end of execution, the agent generates a hash as proof of ByteCode execution.

This hash must be confirmed by multiple agents using FROST multisignature.

At this stage, the blockchain is only a stub. Later on, it will verify the aggregated signature, but for now it can simply act as a final stamp to close the cycle.

    🕸️ What is SAM in I2P?

    SAM (Simple Anonymous Messaging) is a network protocol provided by the I2P router that allows external programs to connect to the I2P network without needing to understand its internal routing architecture.

    📦 Specifically:

    • SAM stands for Simple Anonymous Messaging API
    • It’s a TCP-based interface, usually available on port 7656
    • Programs (like your agent) can connect to I2P through SAM to:
      • Create an I2P identity (destination)
      • Send and receive data within the I2P network
      • Listen for incoming connections

    🛠️ Example Connection

    tcp://127.0.0.1:7656

    🔧 In Rust

    The i2p-sam3 crate is built to work with the SAM protocol. It allows you to:

    • Connect to the I2P SAM gateway
    • Generate I2P keys and set up tunnels
    • Send messages to other I2P destinations
    • Listen for incoming messages

    ✅ Why SAM Matters

    • It simplifies integration of I2P into applications
    • You don’t need to implement a full I2P router or understand its internals
    • It’s the standard way for apps to communicate through I2P securely and anonymously

🔄 4. Detailed Functional Flow

This section describes the step-by-step flow of how a zuz-agent operates from boot to signature broadcast. Each step maps to one or more components defined in section 3.

🧱 1. Agent Bootstrap

  • Component: Startup logic + Config loader
  • At startup, the agent:
    • Loads or auto-creates a config file: agent.yaml (formerly agent.toml).
    • The config includes:
      • peer_id: the agent’s unique ID (can be derived from I2P identity),
      • secret_share: the FROST private share used for signing,
      • ipfs_gateway: I2P-accessible IPFS endpoint (or offline cache),
      • sui_rpc: Devnet RPC endpoint (stub at this stage).

✅ Purpose: prepare local identity and connection settings.

📦 2. Vault Discovery

  • Component: Vault Fetcher (B)
  • Agent continuously checks for new Vaults by:
    • Polling a local file (cids.list) or a remote IPNS endpoint,
    • For each new CID:
      • Downloads .tar archive from IPFS over I2P,
      • Verifies content using vault.json manifest with embedded sha256 checks.

✅ Purpose: find and verify new work units (Vaults).

🎟️ 3. Ticket Handling

  • Components: Lua Sandbox (A), Ticket Executor (C)
  • From the fetched Vault:
    • Extracts and loads ticket.luac — a Lua bytecode script,
    • Executes it in a restricted, deterministic sandbox, passing a read-only:
      • includes Vault metadata, system time, etc.
  • Lua returns Q-structure expressed in YAML as described above:

✅ Purpose: deterministically compute the next vault ticket.

🗳️ 4. Consensus – Step Selection

  • Component: FROST Engine (D), Overlay Network (E)
  • For each step_idx, agents independently:
    • Compute a deterministic leader using:
      min(hash(agent_id || vault_id || ticket_id))
    • The selected leader agent starts a FROST signing round:
      • Sends signing request to other agents over I2P,
      • Collects K-of-L valid signature shares on result_hash.

✅ Purpose: reach decentralized agreement on result correctness.

🔗 5. Sui Blockchain Interaction

  • Component: Stub Blockchain (integration logic only)
  • Once FROST aggregation is complete:
    • Leader builds a PTB (Programmable Transaction Block) calling:
      exec_ticket(vault_id, ticket_id, result_hash)
    • Attaches aggregated signature as sender,
    • Executes a dry-run, then (optionally) submits to the Sui Devnet (stub stage),

✅ Purpose: simulate submission of agreed result (real submission in future phase).

📢 6. Result Broadcast

  • Component: Overlay Network (E), optionally Telemetry (G)
  • After submitting (or dry-running) the step:
    • The leader publishes a signed message:
      {
  "step_idx": 4,
  "tx_digest": "0xabc123..."
}
    • Other agents receive this over I2P,
    • Optionally verify that the transaction is live on-chain (in future versions).

✅ Purpose: ensure all agents are synchronized on step results.

🧠 Additional Notes for Developers

  • All communication is over I2P by default, using the SAM protocol (i2p-sam3 crate).
  • FROST signing is implemented with frost-ed25519, and is transport-agnostic.
  • Lua execution must be sandboxed (mlua + rlimit) and limited to 500 ms and 32 MiB.
  • Logging is controlled via CLI flag -verbose and configured in agent.yaml, with optional OpenTelemetry export.
  • Agents must be testable natively (via VSCode) and in Docker.

🔐 Hash-Chaining Execution Trace in Lua Bytecode

🎯 Purpose

To guarantee that every step of a Lua ticket was executed completely and honestly, we generate a cumulative execution hash as the script runs.

This hash cannot be forged or reconstructed unless every instruction or logical step is actually executed.

⚙️ Step-by-Step Implementation

1. Initialize Execution Hash

At the start of every Lua ticket, we initialize the hash:

local current_hash = sha3_256("ZUZ::execution_start")

2. Update Hash at Each Step

During execution, we update the hash for each meaningful operation. For example:

local step_code = "check_balance:args=1000"
current_hash = sha3_256(current_hash .. step_code)

This can be done:

  • Manually, by inserting code-line annotations at key logic points
  • Or automatically, by wrapping Lua functions or bytecode dispatch in Rust/host

3. Finalize Execution Hash

At the end of execution:

local final_hash = sha3_256(current_hash .. "ZUZ::done")
-- Optional: sign the hash
local signature = sign_with_key(final_hash, private_key)

The final_hash becomes a unique fingerprint of the full execution path. It can be:

  • Broadcast to other agents for FROST signing
  • Included in the YAML output of the ticket
  • Used later for verification

✅ What This Achieves

  • If even one step is skipped, altered, or reordered — the execution_hash will not match
  • If the Lua bytecode is tampered with, the final hash will be invalid
  • Any party can re-run the ticket in a sandbox and verify that the same execution_hash is produced
  • for debugging: Trace logging→ Keep a list of hash contributions for debug/audit purposes

🧠 Where This Runs

  • This logic is implemented inside the Lua ticket itself, or injected into it via the sandbox runtime
  • In Rust, the mlua wrapper can:
    • Intercept key function calls,
    • Track hash updates at each step,
    • Return the execution_hash as part of the final result

🔧 Rust/WASM Version

If the Lua ticket is executed from Rust (as it is in this project), a typical hash chaining flow in Rust might look like:

let mut hash = sha3_256(b"ZUZ::execution_start");

for step in ticket_steps {
    hash = sha3_256(&[&hash, &step_bytes].concat());
}
let final_hash = sha3_256(&[&hash, b"ZUZ::done"].concat());

📦 Integration with FROST

The execution_hash becomes the message for FROST signing.

Only agents who reproduced the same result can contribute a valid signature share.
